Overview

A builder for structured business records.

My mission was to transform the dynamic form builder, used to create records like employee and invoice forms, into a clear and intuitive interface.

The problem

The current builder supported complex business workflows, but the interface was difficult to use.

The goal

Preserve the builder’s flexibility while making it approachable for non-technical users such as operations managers and HR managers.

The Challenge

How can we make a highly flexible form builder feel approachable?

The design needs to balance competing goals:

Simplicity

Reduce the complexity exposed to users.

Flexibility

Supports diverse fields, structures and logic across many business use cases.

Scalability

Grow with the system without becoming overwhelming.

Understanding the problem

Why nobody used the existing builder

We worked closely with a medium-sized manufacturing company, our alpha customer. Generic tools on the market couldn't handle their factory's rigid, format-specific workflows, so they relied on Excel spreadsheets instead, which led to:

  • Duplicated information entry
  • Information scattered across multiple files
  • Manual processes that were difficult to maintain
  • Increased risk of mistakes

The ERP aimed to replace these workflows with structured forms and centralized data, but that vision only worked if creating forms became significantly easier. So I observed and interviewed employees at the factory, mapped their workflows, and examined the existing builder firsthand to understand why it went unused.

Key findings

Patterns pulled from the table and the observation sessions above.

👍

What worked

  • Clean, minimal layouts reduced friction
    Users felt comfortable and moved faster when the workspace stayed uncluttered
  • Clear visual fields
    A visual example reduced hesitation and trial & error, users edited freely
  • Placement indicators for drag-and-drop
    Users could rearrange the form easily without much thought
  • Dynamic multi-column drag-and-drop was the closest reference for the layout flexibility needed
👎

What didn't work

  • Cognitive overload: too many competing elements left users lost.
    "Everything looks the same in a sea of gray, and the workspace feels huge" (HubSpot's canvas)
  • Hidden configuration
    "I wish there was a demonstration of what this does" (Google Forms' multiple-choice grid editor didn't preview the field)
  • Empty states lacking affordance
    "Am I supposed to drag from here? Where do I start?" (canvas didn't signal fields were draggable)

Design

Reducing configuration, increasing visibility (?)

Build on a live canvas

  • The form itself became the main workspace, replacing the old table-based configuration view. Users could see the form as it would appear while they edited it.
  • This removed the save-refresh-repeat workflow and reduced the need to switch between tabs.
  • Clear empty state asks to drag the first element.

Progressive disclosure: Settings only when you need them

  • The old tool showed every option at once.
  • Each field now opens its own panel on selection, and contains only the necessary settings for that field type.

Trade-off:

  • Early concepts used a left panel for fields and a right panel for settings.
  • That created too much visual noise and made the workspace feel crowded.
  • The final compromise used one right-side panel. When field settings open, the fields menu hides.

Start from the real data (Schema-based fields)

  • Fields aren't invented from scratch. They come from the system's existing data model (text, number, date, lists, references)
  • Fields were organized as a tree of the existing model schema
    • For example, when creating an employee form, the user selected the employee schema and then searched or browsed the available employee fields.
  • This kept the builder connected to the system's data structure

Flexible drag-and-drop layouts

  • Users could drag fields into different rows, reorder fields within a row, and place fields between rows.
  • Dividers, tabs, accordions, and tables are elements too, dragged in like any field.
  • What used to require a separate sub-form and manual linking now happens in one motion, on one screen.

Dev handoff & Implementation

<missing title>

Challenges: The drag-and-drop interaction was the most complex part of the implementation. It required significant frontend work and was not an easy task for the development team. I had to push for implementing the drag-and-drop interface, advocating for a better UX despite initial resistance.

Solution: I broke down complex UI tasks into smaller, manageable development sprints. Each sprint, I prioritized the most critical improvement while progressively introducing more advanced front-end features.

Outcomes

  • The interface moved from a table-based configuration tool to a visual builder with drag-and-drop editing and contextual settings
  • The manufacturing company was able to use the daily production input form in their regular workflow
  • We used the builder ourselves inside the team to create our own workflows such as tasks and sprints
  • The creation process became easy and required minimal training
  • Consistency across the ERP: the form builder was built on top of the shared design system, ensuring forms across modules looked and behaved consistently.

Reflection

  • This project taught me that designing a form builder that would serve many different workflows requires deep understanding of the system, mapping all the field types, settings, behaviors, logic rules, and use cases.
  • The biggest lesson was balancing flexibility with simplicity. The builder needed to support uncommon business cases, but the interface could not expose every technical detail at once.
  • That balance shaped the final design: a visual workspace for common tasks, contextual settings for deeper configuration, and progressive disclosure for advanced options.
